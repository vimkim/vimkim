# Repository Guidelines

## Project Structure & Module Organization

This repository is a GitHub profile repository, not an application codebase. The main content lives in `README.md`, which defines the rendered profile page, badges, links, embedded HTML, and the visible 3D contribution image. Generated profile assets are stored in `profile-3d-contrib/*.svg`; treat these as workflow output unless you are deliberately changing generated artifacts. GitHub automation lives in `.github/workflows/profile-3d.yml`. Local assistant/session state such as `.omc/` is ignored and should not be committed.

## Build, Test, and Development Commands

There is no package manager, compile step, or unit-test suite in this repository. Use repository-level checks before committing:

- `git status --short`: confirm only intended files changed.
- `git diff -- README.md AGENTS.md .github/workflows/profile-3d.yml`: review profile and workflow edits.
- `git diff --check`: catch trailing whitespace and conflict markers.

The 3D contribution SVGs are regenerated by the `GitHub-Profile-3D-Contrib` workflow on schedule and through `workflow_dispatch`. Use the GitHub Actions page to run it manually when needed.

## Coding Style & Naming Conventions

Keep Markdown concise and readable. Use ATX headings (`##`), blank lines between sections, and fenced code blocks for command examples. Existing profile sections use `###` headings and short bullet lists; follow that style when adding new link groups. Preserve badge/image Markdown unless changing the rendered profile intentionally. Do not hand-format generated SVG files, and keep generated asset filenames under `profile-3d-contrib/` descriptive and lowercase with hyphens where possible.

## Testing Guidelines

For content changes, preview `README.md` in a GitHub-compatible Markdown renderer and verify that badges, external images, and local SVG references still render. For workflow changes, validate the YAML structure and run the workflow manually from GitHub Actions before relying on the weekly schedule. No coverage requirement applies.

## Commit & Pull Request Guidelines

The history uses automated `generated` commits for workflow output and concise conventional-style messages for manual edits, such as `minor(readme): rename cubrid-jira`. Use short, imperative commit subjects that identify the touched area, for example `docs(readme): add project link`. Pull requests should describe the profile-visible change, mention workflow changes when present, and include screenshots or rendered previews for visual updates.

## Security & Configuration Tips

Do not commit local session files, personal tool state, or secrets. The workflow expects `secrets.PAT`; never hard-code tokens in `.github/workflows/profile-3d.yml`.
